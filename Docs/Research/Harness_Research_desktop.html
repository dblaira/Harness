Chapter 1

The big idea: a harness is the reins

An LLM is a brilliant horse that knows everything about the world and nothing about you.

A large language model is a powerful horse — it can run anywhere, but without reins it throws you or wanders. A harness is the tooling you wrap around it so it goes where you want, with less randomness. Memory, loops, guardrails, evals, and your knowledge graph are all pieces of that harness.

Chapter 2

The four kinds of memory

A bare LLM call is ephemeral — the session is wiped unless you save it.

Working memory — the live context window: prompt + chat history + system prompt. Fast, but gone when the session ends.
Procedural memory — how the agent behaves: rules, habits, skills. Usually a plain SKILL.md.
Semantic memory — durable facts & user profile. Vector DB, fetched with RAG.
Episodic memory — a dated log of past chats & events on a timeline.

Retrieval = RAG, top-K

Never dump the whole DB into the model — too costly, too slow, context tops out (~1M tokens). Text becomes vectors; a similarity search grabs only the top-K most relevant pieces.

"You don't want the entire 2,000 messages. You want the 20 that are exactly relevant."
Chapter 3

The consolidation gate

Why ChatGPT's memory stays short but always current.

Every reply appends to the episodic log, so it explodes. You don't RAG that giant log each turn — you add a gate: after N chats, a cheaper summarizer agent distills events into a handful of durable facts written to semantic memory.

Chapter 4

The agent loop & end-loop guardrails

Agents call tools repeatedly until the job is done. The danger is they never stop.

Loop engineering is deciding when to stop. The end-loop guardrails are the stop conditions: task complete, or a checkpoint where the agent confirms with you first.

His real example: a Claude Code hook that notifies you when the agent is blocked on a permission — so a paused loop never silently wastes 25 minutes.

Chapter 5

LLMOps: the self-improving loop

You can't improve what you can't see. Trace, score, diagnose, ship.

1 · Trace

Record a tree per run: what was asked, what was retrieved, tool calls, latency, tokens. (LangFuse / LangSmith.)

2 · Evaluate

Good & healthy? LLM-as-judge for quality + hard metrics for latency/cost.

3 · Diagnose

Why did it break? Tool too slow? Working memory too big? Retrieval unnecessary?

4 · Ship

Small fix → new prompt / config / retrieval params. Deep bug → fix, re-run, re-trace, re-eval.

Chapter 6 · Your work

The deterministic layer: Understood

The videos cover the probabilistic half. You already built the deterministic half — a knowledge graph that gates the model before it speaks.

The Understood ontology is a two-layer meaning system designed to give an AI personal context without letting it invent facts or pass your beliefs off as universal truth.

Layer 1 — Personal axioms. Your beliefs & causal rules as antecedent → consequent with confidence and evidence links, organized into 13 life domains. Authority: human review only, never silent AI mutation.
Layer 2 — Public guardrails. Formal standards (BFO + Common Core Ontologies, schema.org) so personal beliefs aren't treated as universal truth and stay portable across apps.

The graph is the rails — and it decides before the LLM

This is the load-bearing idea, and it's the deterministic mirror of the video's probabilistic loop: a rule engine reads the graph and decides what's allowed first; the LLM only gets to phrase the result and must cite the rules that fired. It can't make facts up.

How meaning gets into the graph (the steward pipeline)

Nothing enters the canonical graph silently. New meaning starts as a Semantic Candidate and must pass gates before promotion — you are the steward.

Chapter 9

Where to go next

  • Pick the connection model. Harness ↔ ~/.hermes: the macOS app can shell out to understood-kg and read the accepted graph / Fuseki directly; iOS needs a small local service or sync. Decide the boundary first — it shapes everything.
  • Ship graphTrace as the first real feature. "Show me why" on a single answer — render matchedRuleIds + source triple + evidence. It's the soul of the whole system and the most demoable.
  • Port the Seeing Surface dashboard to native SwiftUI by reading the existing /api/summary JSON. Cheap win, immediately useful.
  • Define the agent-routing contract. One job spec → any backend (Claude Code / Codex / Hermes / local). This is what makes Harness the deck and not just another client.
  • Unify the skill / MCP registry. One place to register skills, plugins, MCP servers; agents inherit from it instead of each keeping its own.
  • Centralize authorization (deferred). Authorize once in Harness instead of re-wiring every agent — this is the deck's real plumbing and the frustration that started it all. The original Hermes setup chat lives in Claude.ai (not retrievable on this Mac); when you're ready, reconstruct the model from ~/.hermes config or paste that chat.
  • Optional: wire firecrawl as a native MCP (firecrawl setup mcp) so research like this is one tool call next time.
