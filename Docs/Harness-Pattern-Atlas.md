# Harness Pattern Atlas

Purpose: make Harness visible in recognition order, not engineering order.

Harness should behave like a contrast machine:

- **Harness answer:** grounded in accepted ontology, Adam Pattern step, evidence, and known preferences.
- **Raw LLM answer:** what a general model would likely say without Adam's graph, rules, or pattern.

The value is the difference. The user should be able to feel, "That answer is more me," or, "That answer is generic."

Core rule: **If I can't say, I can't play.** If the current step and evidence cannot be named, Harness should slow execution and locate the pattern.

| "Should I trust the AI answer?" | "Step 1 asks what authority exists. Accepted graph, supporting memory, or model guess?" | Says to verify sources. |

## Prototype Acceptance Test

Ask Harness five real questions. For each one, look at the two outputs:

1. Did Harness name the current step or evidence need?
2. Did the raw answer feel generic by comparison?
3. Did the contrast change what you wanted to build, ask, kill, or observe?
4. Could you point to the difference without technical jargon?
5. Did the app feel more like a mirror than a chat box?

If the answer is no after five real questions, the next branch should not add more machinery. It should redesign the pattern surface.
